# 免费抽抽乐自动化实现方案

## 背景与现状

- 已确认：当前项目已有 `tools/enter_game.py`，可以打开或定位 `BrownDust II` 窗口，使用 `PrintWindow` 截取完整窗口并裁剪客户区。
- 已确认：当前项目已有 `tools/win32_windowpos_click.py`，可以按客户区坐标执行单次点击。
- 已确认：`自动化截图标注/` 中已有主页、主页遮挡弹窗、广场、抽抽乐、确认抽抽乐、抽卡动画、抽卡结算画面的截图和说明。
- 已确认：免费抽抽乐路径中的关键点击点包括主页底部“抽抽乐”、抽抽乐页左侧“服装/装备”、左下“所有免费抽抽乐”、确认弹窗“确认”、动画右上快进、结算页左上返回。
- 推断：脚本应优先做成“状态机”，每轮截图后判断当前页面，再执行当前页面唯一安全动作，而不是按固定时间盲点完整坐标序列。

## 结论

推荐先实现一个独立脚本 `tools/free_gacha.py`，复用现有截图和点击能力，按“识别状态 -> 点击动作 -> 等待下一状态”的状态机完成免费抽抽乐；第一版优先覆盖服装和装备两个入口，遇到未知页面时保存调试截图并停止。

## 需求拆解

### 1. 从主页进入抽抽乐

- 当前现状：`主页.png` 显示底部导航栏左下角有“抽抽乐”，说明文本建议可通过底部文字识别主页。
- 判断：主页识别可以先使用底部导航区域的模板匹配或图像统计；点击点可用客户区相对坐标兜底。
- 修改方案：在 `free_gacha.py` 中增加 `recognize_home()`，识别底部导航区域和“抽抽乐”图标区域；命中后点击底部左一“抽抽乐”。
- 影响范围：新增脚本，不改变现有 `enter_game.py` 的标题页进入逻辑。
- 验证方案：从已在主页的游戏窗口运行脚本，确认能进入抽抽乐界面，并保存 `free_gacha_debug_home.png` / `free_gacha_debug_gacha.png`。

### 2. 处理主页过渡弹窗

- 当前现状：`主页过渡画面.png` 显示主页上方有活动/推送遮罩，说明中指出点击周围暗色区域可以跳过，可能连续出现多个。
- 判断：这是进入主页后的常见阻塞，必须作为独立状态处理。
- 修改方案：增加 `recognize_overlay_popup()`，通过“全屏暗化 + 中央亮矩形”识别弹窗；命中后点击左上或右上暗色空白区域，循环处理直到回到主页或抽抽乐。
- 影响范围：仅影响 `free_gacha.py` 的页面恢复逻辑。
- 验证方案：在弹窗存在时运行脚本，确认能连续关闭弹窗，并最终识别到主页或抽抽乐页。

### 3. 抽取服装免费抽抽乐

- 当前现状：`抽抽乐.png` 显示当前在服装抽抽乐页，左下“所有免费抽抽乐”被红框标注；`确认抽抽乐.png` 显示确认弹窗。
- 判断：左下按钮是第一版最重要识别目标；只在按钮可见时点击，避免误点付费抽取按钮。
- 修改方案：增加 `recognize_gacha_page(kind='costume')` 和 `find_all_free_button()`；识别左侧“服装”选中状态后点击左下“所有免费抽抽乐”；确认弹窗中识别“确认抽抽乐”和“免费X次”后点击确认。
- 影响范围：新增状态 `gacha_costume`、`confirm_free_gacha`。
- 验证方案：在服装抽抽乐页运行脚本，确认能打开确认框、确认后进入动画；如果没有免费次数，脚本应记录 `no_free_gacha=True` 并跳过，不点击付费按钮。

### 4. 抽取装备免费抽抽乐

- 当前现状：`抽抽乐.png` 左侧标注了“装备”切换入口。
- 判断：服装抽完后应返回抽抽乐页，再切换装备，重复“所有免费抽抽乐 -> 确认 -> 动画 -> 结算返回”流程。
- 修改方案：脚本维护 `targets = ['costume', 'gear']`；完成服装后点击左侧装备入口，等待 `gacha_gear` 状态，再执行同样流程。
- 影响范围：新增目标队列和切换动作。
- 验证方案：分别在服装页和装备页启动脚本，确认可以完成当前目标；从主页启动时确认按服装、装备顺序处理。

### 5. 跳过抽卡动画并返回

- 当前现状：`抽卡动画.png` 右上红框为快进按钮；`抽卡结算画面.png` 左上红框为返回上一层按钮。
- 判断：动画和结算页背景很亮，和抽抽乐页差异明显，适合做单独状态识别。
- 修改方案：增加 `recognize_gacha_animation()`，命中后点击右上快进；增加 `recognize_gacha_result()`，命中后点击左上返回，并等待回到抽抽乐页。
- 影响范围：新增状态 `gacha_animation`、`gacha_result`。
- 验证方案：从动画页启动脚本，确认能跳过动画并回到抽抽乐；从结算页启动脚本，确认能返回抽抽乐页。

### 6. 模板与坐标管理

- 当前现状：截图中红框是人为标注，不能直接作为游戏运行时的识别模板。
- 判断：第一版可以同时使用“相对坐标点击 + ROI 图像统计”，但为了后续稳定，应把红框内的关键图标/按钮裁成模板资源。
- 修改方案：新增 `assets/templates/free_gacha/` 保存模板；可另写 `tools/extract_marked_templates.py`，自动检测红色标注框，裁剪框内区域并生成 `templates.json`，记录模板名、来源截图、客户区相对 ROI、默认点击点。
- 影响范围：新增资源目录和可选模板提取工具。
- 验证方案：对现有标注截图运行提取工具，人工检查裁剪结果；再用模板匹配在原截图中回放，确认命中位置与红框一致。

## 执行顺序

1. 先整理模板资源：从现有标注图裁剪“抽抽乐入口”“所有免费抽抽乐”“确认”“快进”“返回”“服装”“装备”。
2. 新增 `tools/free_gacha.py`，复用 `enter_game.capture_client()`、`open_game.find_game_window()`、`win32_windowpos_click.click_client()`。
3. 实现状态识别函数：主页、主页弹窗、抽抽乐页、确认弹窗、动画页、结算页、未知页。
4. 实现状态机主循环：每轮截图、识别、点击、等待、保存必要调试图。
5. 先只跑单步 dry-run：识别当前状态并打印计划点击点，不真实点击。
6. 再启用真实点击，从主页开始完整跑服装免费抽抽乐。
7. 最后加入装备切换和循环终止条件。

## 风险与待确认

- 待确认：游戏分辨率、窗口大小变化时，当前标注坐标是否仍稳定；第一版应要求窗口保持当前截图比例，后续再用模板匹配增强。
- 待确认：“所有免费抽抽乐”在没有免费次数时是否消失、置灰，还是仍可点击后弹出 0 次提示；这会影响跳过逻辑。
- 待确认：服装和装备抽抽乐的确认弹窗文案是否完全一致；如果装备文案不同，需要单独截图补模板。
- 风险：`PrintWindow` 对 Unity 叠层可能有缺失，动画或弹窗识别要保留截图调试输出。
- 风险：现有点击方式会短暂移动窗口到鼠标下方再恢复，执行时不应移动真实鼠标，否则可能影响点击稳定性。
