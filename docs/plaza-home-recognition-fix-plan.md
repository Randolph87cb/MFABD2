# 广场回主页与场景识别修复方案

## 背景与现状

- 已确认：当前游戏实际停留在广场，不是真正主页。
- 已确认：`tools/free_gacha.py` 第一版已经有 `plaza` 状态，但早期试跑曾把广场误判为 `home`，导致反复点击主页底部“抽抽乐”坐标。
- 已确认：后续修正后 dry-run 能把广场识别为 `plaza`，但真实试跑在第一轮截图保存前异常退出，`logs/free_gacha/20260714-221300/events.jsonl` 只有启动事件。
- 已确认：右上角主页按钮点击未产生预期页面切换，可能是点击坐标偏差、Win32 后台点击对该 UI 不生效、或点击后仍需要等待/处理中间弹窗。
- 推断：`H` 键可能是更可靠的“回主页”入口，但需要先验证它能否通过后台消息生效；如果只能前台输入，则不能直接作为默认后台动作。

## 结论

先不要继续跑免费抽抽乐完整流程。推荐先做一个独立的“广场回主页探针”，专门验证三件事：当前状态识别是否准确、右上角主页点击是否真的生效、`H` 键是否能在后台或前台可控条件下回主页。验证通过后再把动作接回 `free_gacha.py`。

## 需求拆解

### 1. 加强场景识别，避免广场和主页混淆

- 当前现状：现有主页识别来自 `enter_game.recognize_home_screen()`，它依赖左上 UI 和小地图区域；广场也有左上小地图/按钮，因此容易被兜底识别为主页。
- 判断：主页和广场必须拆成互斥识别，且广场识别优先级高于主页。
- 修改方案：
  - 新增明确状态：`plaza`、`real_home`、`home_overlay`、`gacha_page`。
  - `plaza` 识别使用广场特征组合：左下虚拟摇杆、右下技能键、右上 `H/O/C` 快捷入口、右侧任务/队伍区域。
  - `real_home` 识别使用主页特征组合：底部导航栏“抽抽乐/伙伴/背包/剧情”等固定图标，以及右侧活动卡片，不再只复用小地图识别。
  - 如果 `plaza` 和 `real_home` 同时命中，判为 `ambiguous`，保存截图并停止，不点击。
- 影响范围：`tools/free_gacha.py` 的 `classify_state()`，后续可抽成 `tools/game_state.py`。
- 验证方案：
  - 用 `自动化截图标注/广场.png`、`主页.png`、`主页过渡画面.png` 做离线分类测试。
  - 用当前实时截图 dry-run，确认广场只输出 `plaza`，不会输出 `home`。

### 2. 把“回主页”改成多策略动作，并要求动作后确认

- 当前现状：脚本点击 `plaza_home = (0.935, 0.055)` 后，没有确认是否离开广场。
- 判断：单次点击不能直接视为成功；必须点击后等待，并连续截图确认状态切换。
- 修改方案：
  - 新增 `return_home_from_plaza()`，内部按策略尝试：
    1. 点击右上角主页按钮区域。
    2. 等待 1-2 秒，截图确认是否变为 `real_home` 或 `home_overlay`。
    3. 如果仍是 `plaza`，尝试 `H` 键策略。
    4. 如果仍失败，保存截图和动作记录，停止并提示用户。
  - 每个策略最多重试 2 次，不做无限循环。
  - 点击坐标不只用单点，改为记录按钮 ROI，并优先点击 ROI 中心；如按钮位置随窗口比例漂移，后续换模板匹配。
- 影响范围：`tools/free_gacha.py` 的 `plaza` 状态处理。
- 验证方案：
  - 从广场运行 `--targets costume --dry-run`，确认计划动作为 `return_home_from_plaza`。
  - 从广场真实运行只执行回主页探针，不进入抽抽乐，确认最终停在主页或遮罩弹窗。

### 3. 验证 `H` 键是否可用于后台回主页

- 当前现状：项目只有鼠标点击封装，没有键盘输入封装。
- 判断：`H` 键可能比点击右上角按钮稳定，但后台键盘消息是否被 Unity 客户端接受未知。
- 修改方案：
  - 新增独立探针 `tools/probe_home_key.py`，只做一件事：在广场状态下发送 `H`，然后截图判断是否回主页。
  - 先测试 Win32 后台消息：`WM_KEYDOWN/WM_KEYUP` 给 `UnityWndClass`。
  - 如果后台消息无效，再测试前台方式：临时激活窗口后用 `pydirectinput.press('h')`。
  - 探针必须把“后台成功/前台成功/均失败”写入日志，不能静默降级。
- 影响范围：新增探针脚本；`free_gacha.py` 只在探针证明可用后接入对应策略。
- 验证方案：
  - 当前在广场时运行探针。
  - 观察日志中的 `method=post_message` 或 `method=foreground_directinput` 和最终状态。
  - 如果前台方式才可行，脚本执行前要明确提示会短暂抢前台。

### 4. 修复截图异常保护

- 当前现状：真实试跑时报 `PIL SystemError: tile cannot extend outside image`，且没有留下步骤截图。
- 判断：这是基础稳定性问题；即使回主页动作修好，也需要先防止转场期间空图/异常裁剪导致脚本崩溃。
- 修改方案：
  - 包装 `capture_client()` 为 `safe_capture_client()`。
  - 校验图片宽高必须大于最小阈值，例如 `1000x600`。
  - 保存图片前调用 `image.load()` 或复制成稳定 RGB 图。
  - 捕获截图/保存异常，最多重试 3 次；仍失败则写 `failure.txt` 并退出。
- 影响范围：`tools/free_gacha.py` 的截图入口和日志保存。
- 验证方案：
  - 在广场切主页转场时运行探针，确认即使截图瞬间异常，也会重试并留下失败截图或原因。

## 执行顺序

1. 先新增离线分类测试入口，对标注截图验证 `plaza`、`real_home`、`home_overlay`、`gacha_page`。
2. 修复 `safe_capture_client()`，保证异常截图不会让脚本直接崩溃。
3. 新增 `tools/probe_return_home.py`，只验证“广场 -> 主页/主页遮罩”，不进入抽抽乐。
4. 在探针里依次验证右上主页点击和 `H` 键。
5. 根据探针结果把成功策略接入 `free_gacha.py`。
6. 再从广场启动完整免费抽抽乐流程。

## 风险与待确认

- 待确认：`H` 键能否通过后台 `PostMessage` 生效；如果不能，需要决定是否允许脚本短暂激活游戏窗口。
- 待确认：右上角主页按钮在不同广场子状态下是否一直是同一个坐标/图标。
- 风险：广场、主页和弹窗都含有大量相似 UI，纯统计识别仍可能误判；长期应把红框标注裁剪成模板资源。
- 风险：如果当前在对话、奖励领取、组队或其他广场子弹窗中，回主页动作可能被遮挡，应先识别并关闭遮挡层。
