## 任务输入摘要

- 最终结果：让《棕色尘埃2》PC 客户端快速狩猎达到够用 MVP，而不是一次性等价 Android 全流程。
- 现有素材：`assets/resource/base` 的 Android pipeline、`assets/resource/pc` 覆盖资源、真实 PC 游戏窗口、截图 fixture。
- 明确约束：优先最小可行改动；流程问题必须加入易错库并增加可重复检查；不要改无关发布流程。
- 完成标准：从 PC 主页或地图页能跑通一次快速狩猎；AP 为 0 时能安全停止；奖励后回到稳定默认地图点；有 harness 和 fixture 证明。
- 产出后动作：提交并推送小步改动，保留后续复刻经验。

## 背景

项目原本以 Android/模拟器流程为主。PC 客户端在坐标、页面结构、主页动画、窗口输入和地图页状态上都不同，直接复用 Android pipeline 容易误判、误点或进入错误分支。快速狩猎被选为第一个 PC MVP，因为它有清晰短链路，且能用真实窗口验证。

## 关键过程

- 先读 `README.md`、`assets/interface.json`、`assets/resource/base`、`assets/resource/pc` 和 `agent/main.py`，确认项目结构和任务入口。
- 对比 `origin/main`、`origin/alpha`、`origin/feat/NT-Suport` 中 PC 适配差异，确认已有 PC 资源不足以完整支撑。
- 建立 PC 控制器和本地启动脚本，优先保证 UI 加载本地新配置：`scripts/start-local-ui.ps1`。
- 用 harness 连接真实 PC 游戏窗口，加载 `base + pc` resource bundle，指定入口节点运行并输出节点序列、识别详情和截图：`scripts/pc_quickhunt_harness.py`。
- 先复刻短链路：`主页快速狩猎入口 -> PC 地图页快速狩猎按钮 -> MAX -> 开始 -> 奖励页领取 -> 回地图 -> 复位到 1.野猪洞穴 -> StopTask`。
- 主页识别从单一颜色/不稳定图像改为组合判断：主页文字 OCR + 右上主页图标，降低动画和活动块造成的误判。
- 遇到误判或错误路径时加入易错库：邮箱页、竞技场页、地图复位失败、AP=0 但仍显示快速狩猎按钮等。
- AP=0 场景显式优先停止，避免先点快速狩猎按钮或回落 Android 地图复位分支。
- 奖励后不直接停止，先点击 PC 地图默认点 `[642,348]`，OCR 确认顶部标题为“野猪洞穴”后停止。
- subagent 只做只读差距审计，主线程负责实现、验证和提交，避免并发改同一批文件。

## 可复用经验

- 不要一开始追求 Android 全流程等价。先选一个可闭环的 PC MVP，再逐段扩展。
- PC 覆盖优先写在 `assets/resource/pc/pipeline/*.json`，尽量不动 base，保留 Android 原逻辑。
- 每个新流程都先明确：入口、稳定页面特征、正向点击、失败停止点、复位点。
- 识别优先组合条件：页面稳定特征 + 关键文字 OCR + 图标或按钮；避免只依赖颜色。
- `StopTask` 是 PC MVP 的安全边界。未知页面、AP=0、复位失败时先停，不让流程掉回 Android 坐标误点。
- 错误样本必须进入易错库，并让检查脚本覆盖；否则下次调阈值或换 ROI 很容易回归。
- harness 输出节点序列比只看 UI log 更有效，能直接证明命中哪个节点、识别到了什么、点了哪里。
- subagent 适合做只读差距审计：base/pc 节点差异、风险点、下一步建议；不要让 subagent 与主线程改同一文件。

## 后续复刻模板

1. 从 `assets/interface.json` 选下一个任务入口，确认是否已挂 PC controller。
2. 读取对应 base pipeline，画出最短可闭环路径。
3. 用真实窗口截图保存正例、负例、失败例。
4. 在 `assets/resource/pc/pipeline` 增加 PC 专用覆盖节点。
5. 增加或更新 corpus manifest 和检查脚本。
6. 用 harness 指定入口跑真实窗口，确认节点序列、OCR 结果和点击坐标。
7. 用启动脚本 `-NoLaunch` 确认 `install` 加载本地新配置。
8. 通过后 `git add`、`git commit`、`git push`，每次只提交一个可验证小步。
